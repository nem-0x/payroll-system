# 学生アルバイト給与計算システム　データ定義書

## 1. 本書の目的

本書は、給与計算システムで扱う全データの入出力定義を明文化するものである。入力データはフォームの物理構造に依存しない「論理項目」として定義し、マスタデータ（従業員、単価、交通費）、照合データ（未提出者検知用）、出力データ（月次集計結果）の項目・型・制約を定義する。フォームの具体的な列定義（物理マッピング）はフォーム確定後に別途作成する。

## 2. 入力データ

### 2.1 データソース一覧

Googleフォームの回答データはスプレッドシートに列として自動格納される。フォームの具体的な列構成は年度ごとに変更される可能性があるため、本書では計算処理に必要な「論理項目」を定義する。フォーム確定後に、論理項目とフォーム列のマッピング定義を別途作成する。

| **データソース** | **対象** | **備考** |
|---|---|---|
| オンラインフォーム | 業務B〜F用 | 業務区分に応じて表示項目が変わる。年度ごとに構成変更あり |
| オフラインフォーム | 業務A（ヘルプ・研修）+ 業務C,D,E | 学校別項目が学校数分繰り返す構造（現在5校）。C,D,Eのコマ数も含む |
| 入退くん | 業務A（通常勤務） | 打刻データ。API経由で取得（Phase 2） |
| その他フォーム | インターン・個別 | 調整金額として処理。本システムの計算対象外 |

> ※ フォーム構成は年度替わりで変更される。論理項目→物理列のマッピング定義はフォーム確定後に作成し、GASのコード内で定数として管理する。

| 3 | 業務日 | date | 全業務共通 | 勤務日 |
| 4 | 業務開始時間 | time | 全業務共通 | 深夜割増の判定には終了時間を使用 |
| 5 | 業務終了時間 | time | 全業務共通 | 22:00以降→深夜割増計算の起点 |
| 6 | 休憩時間（分） | number | 全業務共通 | 分単位の数値 |
| **【業務実績】** | | | | |
| 7 | リアル業務時間（分） | number | A | 自習室勤務。時給1,500円。学校別に取得 |
| 8 | 実施コマ数（面談/講義） | number | B,C,D,E,F | 面談 or 講義の実施コマ。Cは講義 |

## 3. 照合データ: 未提出者検知用

月末締め後の未提出者検知（F1-3）で、日報の提出状況と照合する元データを以下に定義する。

### 3.1 照合元データソース一覧

| **業務** | **照合元データ** | **照合項目** | **備考** |
|---|---|---|---|
| 業務A | 入退くん + シフト表 | 打刻日時、従業員ID | 入退くんAPIで取得 + シフト表SS |
| 業務B | 勤怠管理データ | 日付、従業員No、勤怠ステータス | 業務B固有フォーマット。マッピング要 |
| 業務C | シフト表 | 日付、従業員No、シフト枠 | スプレッドシート |
| 業務D | シフト表 | 日付、従業員No、シフト枠 | スプレッドシート |
| 業務E | 勤怠管理データ | 日付、従業員No、勤怠ステータス | 業務E固有フォーマット。マッピング要 |
| 業務F | 勤怠管理データ | 日付、従業員No、勤怠ステータス | 業務F固有フォーマット。マッピング要 |

### 3.2 勤怠管理データの共通項目

業務B, E, Fの勤怠管理データは業務ごとにフォーマットが異なるが、照合に必要な以下の項目は全業務で揃っている。実装時に業務別のマッピング定義が必要。

| **#** | **項目名** | **型** | **必須/任意** | **備考** |
|---|---|---|---|---|
| 1 | 日付 | date | 必須 | 勤務日 |
| 2 | 従業員No | text | 必須 | 従業員マスタとの照合キー |
| 3 | 従業員名 | text | 参考 | 照合には使用しない（Noで照合） |
| 4 | 勤怠ステータス | text | 必須 | 出勤/欠勤/キャンセル等。業務ごとに値が異なる→マッピング要 |

> ※ 勤怠ステータスの値（出勤/欠勤/キャンセル等）は業務ごとに異なるため、マッピングテーブルを別途定義する。

## 4. マスタデータ

### 4.1 従業員マスタ

| **#** | **項目名** | **型** | **必須/任意** | **備考** |
|---|---|---|---|---|
| 1 | 従業員No | text | 主キー | アルバイト番号（下三桁）と対応 |
| 2 | 氏名 | text | 必須 | |
| 3 | ミョウジ | text | 必須 | 日報との照合用 |
| 4 | 業務区分 | select/複数 | 必須 | A〜F（複数兼務あり） |
| 5 | 在籍ステータス | select | 必須 | 在籍/退職/休職 |
| 6 | Slack User ID | text | 必須 | 通知送信用 |
| 7 | freee従業員ID | text | Phase 3 | freee API連携時に使用 |
| 8 | 入社日 | date | 必須 | |
| 9 | 退社日 | date | 任意 | 退職時に設定 |

> ※ 1人が複数業務を兼務する場合、業務区分は複数値を持つ。
>
> ※ アルバイト番号（下三桁）と従業員Noの対応関係を保持する。

### 4.2 単価マスタ

業務区分ごとの報酬単価・手当単価を管理する。適用開始日/終了日で履歴管理を行い、過去の計算の再検証を可能にする。

| **#** | **項目名** | **型** | **必須/任意** | **備考** |
|---|---|---|---|---|
| 1 | 業務区分 | select | 主キー | A〜F |
| 2 | 基本単価 | number | 必須 | コマ給 or 時給 |
| 3 | 計算単位 | select | 必須 | 時給/コマ給 |
| 4 | コマ所定分数 | number | コマ給時必須 | B:50, C:30, D:20, E:25, F:25 |
| 5 | 当日キャンセル単価 | number | 任意 | B,D,E,Fのみ |
| 6 | 面談手当/コマ手当単価 | number | 任意 | 103円 or 205円 |
| 7 | コマ手当キャンセル適用 | boolean | 必須 | 業務Dのみtrue |
| 8 | 勤務手当単価 | number | 任意 | 103円/日。B,D,E,Fのみ |
| 9 | 保護者MSG手当単価 | number | 任意 | 205円。業務Bのみ |
| 10 | Tel発信手当単価 | number | 任意 | 103円。業務Fのみ |
| 11 | 研修時給 | number | 必須 | 全業務共通 1,226円 |
| 12 | 適用開始日 | date | 必須 | 単価変更の履歴管理用 |
| 13 | 適用終了日 | date | 任意 | null=現在適用中 |

> ※ 「コマ手当キャンセル適用」は業務Dのみtrueとなる例外フラグ。
>
> ※ 新業務追加時は行を追加するだけで対応可能な設計とする。

### 4.3 交通費マスタ

## 5. 出力データ: 月次集計結果

Phase 2で自動算出される月次集計結果の項目定義。スプレッドシート上に個人別・業務別で出力される。

| **#** | **項目名** | **型** | **備考** |
|---|---|---|---|
| 1 | 従業員No | text | 従業員マスタ参照 |
| 2 | 氏名 | text | |
| 3 | 対象年月 | text | YYYY-MM |
| 4 | 業務区分 | text | 複数業務兼務時は業務ごとに行を分ける |
| 5 | 基本報酬（通常コマ） | number | コマ給×実施コマ数 or 時給×分数 |
| 6 | 基本報酬（キャンセルコマ） | number | キャンセル単価×コマ数 |
| 7 | 面談手当/コマ手当 | number | 業務別の手当合計 |
| 8 | 勤務手当 | number | 103円×面談実施日数 |
| 9 | 保護者MSG手当 | number | 業務Bのみ |
| 10 | Tel発信手当 | number | 業務Fのみ |
| 11 | 研修・事務作業報酬 | number | 1,226円÷60×分数 |
| 12 | 深夜割増 | number | コマ給ベース（労働者有利）。10分超はアラートフラグ |
| 13 | 交通費（通常） | number | 登録額×出勤日数 |
| 14 | 交通費（例外） | number | 日報申請分 |
| 15 | 休業手当 | number | 会社都合キャンセル分 |
| 16 | プロインセンティブ | number | 対象者のみ 1,000円 |
| 17 | 調整金額 | number | コンプラ+新人研修+有給+紹介料の一括 |
| 18 | 月次総支給額（税前） | number | ROUNDUP(①〜⑰の合計) |
| 19 | 深夜アラートフラグ | boolean | 22時以降10分超の場合true |
| 20 | 確定ステータス | select | 未確定/確認済/確定/freee送信済 |
| 21 | 修正履歴 | text | 手動修正時の理由・日時を記録 |

> ※ 複数業務を兼務する場合、#4〜12は業務ごとに行を分け、#13以降は個人単位で1行にまとめる。
>
> ※ #18の月次総支給額は全業務合算後にROUNDUP（1円未満切り上げ）を1回適用する。
>
> ※ #20の確定ステータスは業務フロー上の進行管理に使用する。「確定」後は原則として修正不可（修正する場合は#21に理由を記録）。

## 6. データ間の関連

1. 日報フォームのアルバイト番号 → 従業員マスタの従業員No で照合する。氏名（ミョウジ）は補助的な確認用。
2. 従業員マスタの業務区分 → 単価マスタの業務区分 で適用単価を決定する。
3. 従業員マスタの従業員No → 交通費マスタの従業員No で1日あたり交通費を取得する。
4. 未提出者検知では、各業務の照合元データの従業員No/ID と 日報のアルバイト番号 を従業員マスタ経由で紐付ける。
5. 月次集計結果の従業員No → freee従業員ID のマッピングはPhase 3のfreee API連携時に使用する。

## 7. 補足事項

1. フォームの物理列→論理項目のマッピング定義は、フォーム構成が確定した時点で別途作成する。年度替わりでフォームが変更された場合はマッピング定義のみを更新し、計算ロジック本体は変更不要な設計とする。
2. インターンフォーム・個別フォームのデータは本システムの計算対象外とし、調整金額として手動入力する。
3. オフラインフォームの学校数が変動する場合、フォーム構造の修正とマッピング定義の更新が必要。
4. 勤怠管理データのフォーマットマッピング定義はPhase 1の実装開始時に業務B, E, Fそれぞれについて作成する。
5. プロインセンティブの対象者リスト、休業手当のシフトキャンセルデータ、調整金額の各項目（コンプラテスト・新人研修・有給・紹介料）は、スプレッドシート上の手動管理データとして別途取り込む。
