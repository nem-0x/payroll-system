# 学生アルバイト給与計算システム　要件定義書（To-Be）

## 1. 本書の目的・背景

本書は、学生アルバイト給与計算業務の自動化・効率化に向けたシステム要件を定義するものである。現状整理書（As-Is）および計算ロジック仕様書で整理した業務実態に基づき、目指すべき姿（To-Be）と実現手段を示す。

現状の課題として、月次給与計算に3日以上を要していること、日報不備の検知・修正に多大な工数がかかっていること、集計からfreee入力まで1名で属人的に運用していることが挙げられる。本システムの導入により、これらの課題を段階的に解消し、正確かつ効率的な給与計算体制を構築する。

## 2. スコープ定義

| **領域** | **スコープ** | **備考** |
|---|---|---|
| 日報不備の自動検知 | 対象（フェーズ1） | フォーム送信時に即時バリデーション |
| 未提出者への自動リマインド | 対象（フェーズ1） | Slack通知（月次一括） |
| 給与計算の自動集計 | 対象（フェーズ2） | 計算ロジック仕様書に基づく自動算出 |
| 入退くんデータ連携 | 対象（フェーズ2） | API経由で自動取得 |
| freee人事労務への自動連携 | 対象（フェーズ3） | freee API経由で勤怠データPUT |
| 給与明細の発行 | 対象外 | freee上の既存機能を継続利用 |
| 年末調整 | 対象外 | freee上の既存機能を継続利用 |
| 承認フローのシステム化 | 対象外（現状維持） | 業務Aの教室長確認は従来通り運用 |

## 3. フェーズ計画

段階的に導入し、各フェーズで効果を確認しながら次フェーズに進む。

| **フェーズ** | **内容** | **主要機能** | **目標時期** |
|---|---|---|---|
| Phase 1 | 不備検知・リマインド | 日報バリデーション／未提出者検知（月次一括）／Slack通知／深夜割増アラート（10分超） | 2〜3ヶ月以内 |
| Phase 2 | 集計自動化 | 業務A〜F計算ロジック実装／入退くんAPI連携／共通手当・調整金額の自動集計／月次総支給額の自動算出 | Phase 1完了後 3〜4ヶ月 |
| Phase 3 | freee連携 | freee人事労務API連携／勤怠データ自動PUT／差分チェック・整合性検証 | Phase 2完了後 2〜3ヶ月 |

## 4. 技術スタック

既存のGoogle Workspace環境とGAS開発基盤を最大限に活用する方針とする。

| **技術要素** | **用途** | **選定理由** |
|---|---|---|
| Google Apps Script | メイン処理基盤 | 既存のGoogleフォーム・スプレッドシートとシームレスに連携。開発・運用実績あり |
| Google Spreadsheet | データストア・計算エンジン | マスタ管理、集計結果の格納、人間による確認・修正のUI |
| Google Forms | 日報入力 | 既存の入力フローを維持。フォーム送信トリガーでバリデーション起動 |
| 入退くん API | 業務Aの勤怠データ取得 | 手動エクスポートを廃止し、API経由で自動取得 |
| Slack API | リマインド・アラート通知 | 全アルバイトが参加済み。不備通知・未提出リマインド・深夜アラートに使用 |
| freee人事労務 API | 勤怠データPUT（Phase 3） | 手入力を代替。OAuth2認証は既存実装を流用 |

> ※ GASの6分実行制限を考慮し、100名弱の計算処理が制限内に収まる設計とする。超過する場合はバッチ分割で対応。

## 5. 機能要件

### 5.1 Phase 1: 不備検知・リマインド

日報の品質向上と提出率改善を目的とする。集計作業の前段階で不備を減らすことで、Phase 2以降の集計自動化の効果を最大化する。

| **ID** | **機能名** | **詳細** | **トリガー** |
|---|---|---|---|
| F1-1 | 日報バリデーション | オンラインフォーム（B〜F）・オフラインフォーム（A例外）それぞれに対し送信時に以下をチェック: 開始時刻 < 終了時刻／業務区分とコマ数の整合性／必須項目の入力漏れ／時間と業務内容の矛盾検出 | Googleフォーム送信時（onFormSubmit） |
| F1-2 | 不備通知 | バリデーションエラー検出時に提出者へSlackで通知。エラー内容の具体的な説明・再提出を促すメッセージ | F1-1でエラー検出時 |
| F1-3 | 未提出者検知 | 月末締め後、業務ごとに元データと日報を照合。業務A: 入退くん打刻+シフト表 vs 日報／業務C,D: シフト表 vs 日報／業務B,E,F: 勤怠管理データ vs 日報（日付・従業員No・勤怠ステータスで照合。業務ごとにフォーマットが異なるためマッピング要）。勤務実績があるのに日報未提出の者を抽出 | 月次（締め日後に手動 or 自動実行） |
| F1-4 | 未提出リマインド | 未提出者へSlackで通知。未提出の対象日一覧を提示・再提出期限を明記 | F1-3の検知結果に基づく |
| F1-5 | 深夜割増アラート | 日報の終了時刻が22:10以降の場合、担当者にSlackでアラートを出力（22時以降10分超の確認用） | Googleフォーム送信時 |
| F1-6 | 管理ダッシュボード | 提出状況・不備一覧をスプレッドシート上で可視化。未提出者リスト（フォーム別）・不備ステータス管理・リマインド送信履歴 | スプレッドシート（常時参照可） |

### 5.2 Phase 2: 集計自動化

計算ロジック仕様書に基づき、業務A〜Fの給与計算を自動化する。人間による最終確認ステップは残す。

| **ID** | **機能名** | **詳細** | **備考** |
|---|---|---|---|
| F2-1 | 入退くんAPI連携 | 業務Aの出退勤データをAPI経由で自動取得。スプレッドシートに自動格納・日報データとの統合処理 | Phase 2で入退くんAPI仕様の調査が必要 |
| F2-2 | 業務別給与自動計算 | 計算ロジック仕様書に基づき業務A〜Fの給与を自動算出。基本報酬（時給/コマ給）・業務固有手当・深夜割増（コマ給ベース・労働者有利）・端数は月合計時に切り上げ | 業務Dのコマ手当例外（キャンセルコマも対象）に注意 |
| F2-3 | 共通手当・交通費の自動計算 | 交通費: マスタ登録額×出勤日数（オンライン日除く）、例外交通費は日報申請額を加算。休業手当: SS上のデータから算出。プロインセンティブ: 対象者リスト照合 | 交通費重複時は単発優先 |
| F2-4 | 調整金額の取り込み | コンプラテスト手当・新人研修手当・有給・紹介料をスプレッドシート/別システムから取り込み一括計上 | 紹介料は都度単価指定 |
| F2-5 | 月次総支給額算出 | 全項目を合算し、ROUNDUPで月次総支給額を確定。個人別サマリーシート出力・前月比較・異常値検出 | 人間による最終確認ステップを設ける |
| F2-6 | 確認・修正UI | 計算結果をスプレッドシート上で確認・手動修正可能に。修正履歴の記録・修正理由の入力欄 | 自動計算後の人的チェックを担保 |

### 5.3 Phase 3: freee連携

集計スプレッドシートで確定した給与データをfreee人事労務APIで自動連携し、手入力を廃止する。

| **ID** | **機能名** | **詳細** | **備考** |
|---|---|---|---|
| F3-1 | freee API連携 | 確定した勤怠データをfreee人事労務APIでPUT。従業員IDマッピング・勤怠項目のマッピング・OAuth2認証（既存実装を流用） | freee APIの勤怠PUT対応範囲の事前調査が必須 |
| F3-2 | 差分チェック | API送信前にスプレッドシートの確定値とfreee側の既存データを比較し、差分を表示。上書き前の確認フロー | 誤送信防止のためのセーフガード |
| F3-3 | 送信結果ログ | API送信の成功/失敗を記録。エラー時のリトライ機構・送信履歴の管理 | |

> ※ Phase 3着手前にfreee人事労務APIの勤怠データPUT対応範囲を調査する必要がある。特に年末調整関連項目のAPI対応可否は要確認。

## 6. 非機能要件

| **カテゴリ** | **要件** | **詳細** |
|---|---|---|
| 正確性 | 計算結果の正確性保証 | 手動計算との突合テストを実施。全業務パターンでの検証を必須とする |
| 可監査性 | 計算過程の追跡可能性 | 計算の中間値・適用ロジックをログとして残し、後から検証可能にする |
| 運用性 | 属人化の排除 | 担当者不在時にも別の人が運用できるよう、操作手順書を整備する |
| 保守性 | 給与体系変更への対応 | 単価・手当の変更をマスタデータの修正のみで対応可能にする。コードの変更を最小化 |
| 拡張性 | 新業務パターンの追加 | 4月追加予定の2業務を含め、業務区分の追加がマスタ追加で対応可能な設計とする |
| セキュリティ | 個人情報の保護 | 給与データへのアクセス権限を最小限に。スプレッドシートの共有範囲を制限 |
| 性能 | GAS実行時間制限への対応 | 100名弱の処理をGASの6分制限内に収める設計。必要に応じてバッチ分割 |
| 信頼性 | エラーハンドリング | API障害・データ不整合時の自動リトライとアラート。処理途中での中断に対する復旧手順 |

## 7. データフロー

| **入力元** | **データ内容** | **処理** | **出力先** |
|---|---|---|---|
| Googleフォーム | 日報データ（B〜F+A例外） | バリデーション→集計SS | 集計スプレッドシート |
| 入退くん API | 出退勤データ（業務A） | API取得→集計SS / 未提出検知照合 | 集計スプレッドシート |
| シフト表SS | 勤務予定（業務A,C,D） | 日報との照合→未提出者検知 | 管理ダッシュボード |
| 勤怠管理データ | 勤怠実績（業務B,E,F） | 業務別にフォーマット異→マッピング→照合 | 管理ダッシュボード |
| マスタSS（交通費） | 1日あたり登録額 | 出勤日数と掛け合わせ | 集計スプレッドシート |
| マスタSS（従業員） | 業務区分・時給・Slack ID | 計算ロジック適用の基準 | （参照のみ） |
| 調整金額SS | コンプラ・新人研修・紹介料 | 金額をそのまま取り込み | 集計スプレッドシート |
| 別システム | 有給金額 | 金額転記のみ | 集計スプレッドシート |
| 集計スプレッドシート | 確定済み給与データ | freee API PUT（Phase 3） | freee人事労務 |

## 8. マスタデータ設計

給与体系の変更にコード修正なしで対応するため、以下のマスタデータをスプレッドシートで管理する。

| **マスタ名** | **主要項目** | **更新頻度・管理方法** |
|---|---|---|
| 従業員マスタ | 氏名、業務区分、基本時給/コマ給、Slack User ID、freee従業員ID | 入退社時に更新。スプレッドシートで管理 |
| 交通費マスタ | 従業員ID、1日あたり交通費登録額 | 住所変更・通勤経路変更時に更新 |
| 単価マスタ | 業務区分、基本コマ給、キャンセル単価、各種手当単価、コマ所定分数 | 給与体系変更時に更新。変更履歴を保持 |
| プロインセン対象者リスト | 対象月、従業員ID | 毎月更新 |
| 勤務カレンダー | 営業日、祝日、特別休業日 | 年次で設定。未提出検知の基準として使用 |

> ※ 単価マスタに変更履歴を持たせることで、過去の計算の再検証が可能となる。

## 9. To-Be業務フロー（Phase 2完了時点）

Phase 2完了後の目標業務フローを以下に示す。現状の6ステップのうち、ステップ3〜5が大幅に自動化される。

1. アルバイトがGoogleフォームで日報を提出。オンラインフォーム（業務B〜F）とオフラインフォーム（業務Aのヘルプ・研修）の2種類。業務Aの通常勤務は入退くんで打刻。インターン等の個別フォームは調整金額として処理。
2. GASがフォーム送信時に即時バリデーションを実行。不備があれば提出者にSlackで通知し、再提出を促す。
3. 教室長が業務Aの日報を一次確認（現状維持）。
4. 月末締め後、GASが未提出者を一括検知し、Slackで未提出リマインドを送信。
5. GASが全データを自動集計し、業務別・個人別の給与計算結果をスプレッドシートに出力。深夜割増10分超アラートがあればSlackで担当者に通知。
6. 担当者がスプレッドシート上で計算結果を確認・必要に応じ修正。確定ボタンで給与データをロック。
7. freee人事労務にデータ入力（Phase 2時点では手入力、Phase 3でAPI自動化）。
8. freeeで給与明細を発行。

**目標効果:** 現状3日以上かかっている集計作業を、確認・修正を含めて半日〜1日程度に短縮する。

## 10. 制約事項・前提条件

1. Googleフォームは現行の2種類（オンライン/オフライン）をベースとする。バリデーション強化のために項目追加・選択肢修正が必要になる可能性がある。インターン・個別フォームのデータは調整金額として処理する。
2. 通知はSlack APIを使用する。全アルバイトがSlackに参加済みであることを前提とする。
3. 未提出者検知の照合元データ（シフト表・勤怠管理データ）は業務ごとにフォーマットが異なる。Phase 1でデータ項目のマッピング定義が必要。照合に必要な項目（日付・従業員No・勤怠ステータス）は全業務で揃っている。
4. 入退くんAPIの仕様・認証方式はPhase 2開始前に調査が必要。
5. freee人事労務APIの勤怠PUT対応範囲はPhase 3開始前に調査が必要。特に年末調整に影響する項目のAPI対応可否を確認する。
6. 4月追加予定の2業務パターンは、詳細判明次第マスタデータに追加して対応する。
7. 開発・運用は情シス担当1名体制。各フェーズの開発と並行して通常業務も継続するため、スケジュールには余裕を持たせる。

## 11. 成功基準

- **Phase 1:** 日報の不備率が50%以上減少し、未提出による締め後再計算がゼロになること。
- **Phase 2:** 月次集計作業が半日〜1日以内に完了し、計算精度が手動計算と100%一致すること。
- **Phase 3:** freeeへの手入力がゼロになり、API送信の成功率が99%以上であること。
